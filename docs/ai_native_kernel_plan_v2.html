1. ภาพรวมสถาปัตยกรรมและขอบเขต MVP

การพัฒนา AI-Native Kernel ใน Phase 1 จะใช้แนวทาง Hybrid-Companion Model โดยทำงานร่วมกับ Linux Kernel ผ่านระบบ eBPF และ LSM เพื่อดักจับพฤติกรรมในระดับริงศูนย์ (Kernel Space) และส่งต่อการควบคุมมายังระบบหลักที่พัฒนาขึ้นด้วยภาษา Rust

หลักการออกแบบ (Design Principles):

  • Memory Safety Without GC: Rust ป้องกัน Use-After-Free / Buffer Overflow โดยไม่ต้องมี Garbage Collector
  • Async-First: ใช้ Tokio runtime รองรับ Phase 1 ที่ 10 agents และออกแบบให้ขยายไปถึง 1,000+ agents ในเฟสถัดไป
  • Fail-Closed for Security, Graceful Degradation Elsewhere: security decision paths ต้อง default = DENY; ส่วน subsystem อื่นลดระดับบริการได้เมื่อปลอดภัย
  • Observability Built-In: Metrics + Tracing ฝังอยู่ในทุก hot path ตั้งแต่วันแรก

Module 3: Compute Scheduler & Decision Matrix

คัดเลือกหน่วยประมวลผลที่ประหยัดพลังงานและได้ผลลัพธ์เร็วที่สุด โดยใช้ Cost Function:

Cost = (w₁ × Latency) + (w₂ × Power) + (w₃ × FinancialCost)

Adaptive Weights (3 โหมด):

  • Battery Mode (Laptop/Edge): w₂ (Power) สูง → ลดการใช้ GPU
  • Throughput Mode (Server): w₁ (Latency) สูง → เลือก GPU ก่อน
  • Cost Mode (Cloud Burst): w₃ (FinancialCost) สูง → เลือก local NPU ก่อน
  • Adaptive learning: ปรับ weight ทุก 100 runs ด้วย EWMA

4.1 สถาปัตยกรรมภูมิคุ้มกัน (Immune System Architecture)

ระบบ AI-Native Kernel ถูกออกแบบให้มีกลไกป้องกันตัวเองแบบสิ่งมีชีวิต โดยใช้ AI Agents ทำหน้าที่เป็นเม็ดเลือดขาว (White Blood Cell Agents):

  • Macrophage Agents: เดินลาดตระเวน (Patrol) เพื่อทำ Garbage Collection โดยการตรวจจับและทำลาย Context ที่หมดอายุ หรือ Token ขยะใน IntentBus
  • T-Cell Agents (หน่วยพิฆาต): เฝ้าระวังพฤติกรรมน่าสงสัยระดับ Syscall ผ่าน eBPF หากพบโปรเซสหรือ Agent ที่ทำตัวเป็นภัยคุกคาม จะสั่งระงับและหยุดการทำงาน (Kill) ทันที
  • B-Cell Agents (หน่วยสร้างเกราะป้องกัน): เรียนรู้รูปแบบการโจมตีใหม่ๆ และสร้างกฎ LSM Policy (Antibody) แบบไดนามิก เพื่ออัปเดต Policy Engine อัตโนมัติ ป้องกันการโจมตีแบบเดิม
  • Cytokine Signals: เมื่อพบภัยคุกคามร้ายแรง Agent สามารถกระจายสัญญาณเตือน (Critical IntentEvent) ผ่าน Bus เพื่อส่งให้ Swarm ของ Agent มารุมจัดการปัญหาที่จุดเดียว

Testing Invariants (Property Tests):

  • Agent Scheduler: Running + Ready + Waiting == total active agents
  • Capability Token: expired tokens always rejected (constant_time_eq)
  • Context Memory: Hot→Warm→Cold→Warm→Hot round-trip lossless
  • Cost Function: selected target is global minimum for given weights

H1 cgroup-scoped allow-list LSM gate เป็น PID block-list (default-ALLOW) — ทางแก้ชั่วคราวจาก d728095d ไม่ใช่ design ที่ตั้งใจ รัน agent ทุกตัวใน cgroup เฉพาะ แล้วให้ eBPF เช็ค cgroup id แทน PID — ได้ default-DENY เต็มรูปแบบเฉพาะโลกของ agent โดยไม่แตะ process ของ host (host ไม่มีทางค้าง) ✅ implemented + validated end-to-end บน privileged host 10 ก.ค. 2026 (opt-in ผ่าน lsm.agent_cgroup_path; probe ยืนยัน default-DENY ใน agent cgroup และ host ไม่กระทบ ผ่าน scripts/validate-ebpf-attach.sh) Phase 1.5
H2 Agent identity ที่ปลอมไม่ได้ การผูก agent กับ PID ตรง ๆ มีช่องโหว่ PID reuse — process ใหม่อาจสวมสิทธิ์ PID เดิม ใช้ pidfd หรือ tuple (start_time, pid) ใน BPF map เพื่อผูก capability token กับ process instance จริง ✅ implemented + validated end-to-end 10 ก.ค. 2026 (allow-list เก็บ start ticks จาก /proc/<pid>/stat เทียบกับ start_boottime ใน kernel; ทั้งขา allow และ deny ผ่าน tests/privileged_h1_h2.rs ที่ขับ authorize flow จริงกับ child process จริงบน privileged host) Phase 1.5
H3 Intent → Scope compiler Semantic gap: kernel เห็นแค่ syscall แต่ไม่รู้ intent ที่ประกาศไว้ — ชิ้นส่วนมีครบ (intent-bus, policy, LSM) แต่ยังไม่มีตัวแปลตรงกลาง Pipeline แปลง intent เป็น scope แคบที่ตรวจได้เชิงกลไก (เช่น intent อ่านโปรเจกต์ X → token เปิดอ่านเฉพาะ path นั้น) แล้ว push ลง BPF map ก่อน agent เริ่มงาน ✅ implemented + validated end-to-end 10 ก.ค. 2026 (เฟสแรก: operation-class flags + path prefix ตรวจใน kernel ด้วย bpf_d_path; token = grant, intent = narrow ได้เท่านั้น; tests/privileged_h1_h2.rs ยืนยัน in-scope allow / out-of-scope deny / exec deny บน kernel จริง) Phase 2
H4 Real-time token revocation เมื่อ TCell สั่ง quarantine การเพิกถอนสิทธิ์ยังผ่าน userspace round-trip — agent อาจยิง syscall ได้อีกหลายพันครั้งก่อนโดนตัด Path ตรงจาก immune-system decision → เขียน BPF map ทันที ไม่รอ policy engine round-trip ✅ implemented ก.ค. 2026 (TCell quarantine/kill → deny_pid เขียน blocked_pids map แบบ synchronous ก่อน audit/broadcast แล้วค่อยเพิกถอน token; unit test ยืนยันตัดที่ kernel + revoke token) Phase 2
H5 VRAM tier จริง gpu_pool.rs ยังเป็น stub (TODO: real cudaMemcpy) — ชั้น GPU ของ context paging ยังไม่ต่อจริง Implement cudaMemcpy/hipMemcpy device↔host จริง — ต้องมีเครื่อง CUDA/ROCm สำหรับทดสอบ (ตรงกับ p2a ใน Gantt) ✅ implemented ก.ค. 2026 (migrate_to_cpu ทำ DtoH จริงรักษาข้อมูล + เพิ่ม migrate_to_gpu คู่กัน HtoD; round-trip test ในโหมด simulation ผ่าน; real-GPU byte-for-byte round-trip test พร้อมรันบนเครื่องที่มี CUDA — ⚠️ ยังรอ hardware validation) Phase 2
H6 P2P mesh mutual auth + integrity mesh transport เป็น raw TCP + JSON plaintext ไม่ auth peer — ใครต่อ TCP มาถึง port ก็ยัด record ปลอม/ปลอมเป็น node trust สูง/ดักอ่าน/replay ได้ ขัด Zero-Trust ของโปรเจกต์เอง เซ็นทุกข้อความด้วย HMAC-SHA256 (pre-shared key ต่อ mesh) + ปฏิเสธ replay (timestamp window + nonce dedup) ✅ implemented + validated ก.ค. 2026 (module mesh_auth: seal/open; wire ทุกจุดใน p2p_mesh; companion fail closed ตอน boot ถ้าเปิด mesh โดยไม่ตั้ง key; E2E ผ่าน TCP loopback ยืนยัน matching-key เชื่อมได้/wrong-key ถูกปฏิเสธ) Phase 4
H7 P2P mesh confidentiality (mTLS) H6 ให้ integrity + authenticity + anti-replay แต่ payload ยังวิ่ง plaintext บนสาย ดักอ่าน (passive eavesdropping) และ active MITM แอบอ่านได้ เข้ารหัสสายด้วย TLS 1.3 โดยไม่ต้องมี PKI: derive cert/key แบบ deterministic จาก PSK เดียวกับ H6 แล้ว pin peer cert ให้ตรง identity นั้น (module mesh_tls: rustls custom verifier + rcgen; wrap TLS ใน start_listener/connect_to_peer) ✅ implemented + validated ก.ค. 2026 (E2E: matching-PSK handshake ผ่าน, wrong-PSK ถูกปฏิเสธที่ชั้น TLS; operator จัดการ secret เดียวเหมือนเดิม) Phase 4
H8 Capability-scoped skill manifests "แบ่งความถนัด" ของ agent (specialization) ยังเป็นแค่แนวคิด — ไม่มีวิธีประกาศ specialist พร้อมขอบเขตสิทธิ์ที่ kernel บังคับได้ skill.md-style manifest (frontmatter TOML) ประกาศพร้อมกันทั้ง routing (description), placement (model/compute) และ capability scope ([capabilities]) — manifest เป็น "สัญญาที่ kernel ตรวจ" ไม่ใช่คำแนะนำ; scope ที่ประกาศ compile ผ่าน H3 (narrow เท่านั้น) แล้ว kernel บังคับ (module skill: parser + registry + routing) ✅ implemented + validated ก.ค. 2026 (unit: parse/narrow/route; E2E บน kernel จริง: authorize agent จาก skill manifest → in-scope เปิดได้/out-of-scope+exec โดนปฏิเสธตามที่ skill ประกาศ) Phase 3

ลำดับที่แนะนำ: H1 ก่อน เพราะเป็นช่องโหว่เชิงสถาปัตยกรรมที่ขัดกับ security model ของโปรเจกต์เอง และ cgroup scoping จะปูทางให้ H2 (identity ผูกกับ cgroup) และ H3 (scope ผูกกับ intent) ต่อได้ทันที

10. Phase 1 MVP Success Criteria

Phase 1 จะถือว่า "สำเร็จ" เมื่อตรงเกณฑ์ทั้งหมด โดยแยกสิ่งที่ยืนยันได้ใน repo ออกจากสิ่งที่ยังต้อง re-validate บน host จริง:

  • ✅ Host-side prototype build/check/clippy/test ผ่านบน pinned toolchain และ cross-crate pipeline ทำงานครบ
  • ✅ Agent runtime รองรับ 10 concurrent agents, spawn P99 < 500 µs ตาม perf-budget tests
  • ✅ Capability/LSM policy path ตัดสินใจภายใน 1 ms ใน host-side validation path
  • ✅ Audit Log บันทึก security decisions แบบ append-only พร้อม hash-chain validation
  • ✅ ผ่าน chaos / pipeline / integration tests สำหรับ Failure Domains ที่อยู่ใน workspace ปัจจุบัน
  • ✅ Qdrant-backed ignored tests (4 ตัว: semantic store ×3 + SFS ×1) ผ่านกับ Qdrant จริง 1.18.2 (podman, 10 ก.ค. 2026) — semantic path ยืนยันแล้ว
  • ✅ eBPF/LSM real attach พร้อม fallback disabled ผ่านการ validate บน privileged host (kernel 7.0.0-27, 10 ก.ค. 2026) รวม H1 cgroup-scoped default-DENY end-to-end
  • ✅ cargo clippy --all-targets --all-features -- -D warnings และ cargo audit เป็น baseline validation ของ repo ปัจจุบัน
